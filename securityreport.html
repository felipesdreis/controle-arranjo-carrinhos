Critical jsPDF HTML Injection — Arbitrary JS Execution in New Window Paths Dep. Vuln (A06)
GHSA-wfv2-pwc8-crg5 CVSS 9.6 GHSA-f8cm-6447-x5h2 LFI/Path Traversal
package.json · html2pdf.js@0.10.2 → jspdf@≤4.2.0

jsPDF ≤ 4.2.0 (pulled in as a transitive dependency of html2pdf.js@0.10.2) contains an HTML injection vulnerability in its "New Window" output paths (CVSS 9.6). A crafted filename, congregation name, or brother name saved to a PDF could execute arbitrary JavaScript in the context of the opened window. The same version also carries a Local File Inclusion / Path Traversal bug that allows reading arbitrary files when the library processes attacker-controlled filenames. This app passes filename: `testemunho_${weekStart}.pdf` and DOM content including user-controlled data (congregation names, brother names) directly into html2pdf.js.

Fix: Upgrade html2pdf.js to ≥ 0.14.0. The CLAUDE.md note "não atualizar" (don't update) cites "inoffensive transitive vulns" — this assessment is incorrect; the jsPDF CVEs are actively exploitable. Run npm install html2pdf.js@^0.14.0 and validate PDF output. All jsPDF CVEs ≤ 4.2.0 are patched in 4.3+.
Critical html2pdf.js XSS Vulnerability Dep. Vuln (A06)
GHSA-w8x4-x68c-m6fc CWE-79
package.json · html2pdf.js@0.10.2

html2pdf.js@0.10.2 itself carries a cross-site scripting vulnerability (GHSA-w8x4-x68c-m6fc). The library converts arbitrary HTML DOM nodes to PDF; if any unescaped user content reaches the DOM subtree captured for PDF generation, it could be exploited as an XSS vector. The report page (Report.jsx) passes reportRef.current — a DOM node containing brother names, congregation name, and location strings from the database — directly to html2pdf().from(reportRef.current).

Fix: Upgrade to html2pdf.js@^0.14.0. React's JSX engine auto-escapes content rendered via JSX, providing partial mitigation; however the library-level fix is required to eliminate the vulnerability entirely.

jsPDF ≤ 4.2.0 does not validate BMP and GIF image dimensions before allocating memory, allowing a crafted image (e.g., embedded in the DOM captured for PDF) to cause unbounded memory allocation and crash the client browser tab.

High Raw Supabase / PostgREST Error Messages Exposed to Users Error Handling (A09)
src/store/useAppStore.js:49, 59 · src/pages/Brothers.jsx:74 · src/App.jsx:109

Every catch block in useAppStore.js calls set({ error: err.message }), propagating raw error messages (which include PostgREST/Supabase internal messages like "permission denied for table brothers", "JWT expired", or database constraint names) directly to the UI. Pages render {error} verbatim. Additionally, AdminUsersPage.jsx calls alert(`Erro: ${err.message}`) on failures, exposing raw errors in alert dialogs. This violates OWASP A09 and can reveal schema details, policy names, and internal state.

Fix: Map Supabase error codes to user-friendly messages at the store layer (similar to the existing mapErrorMessage() in src/api/auth.js). Create a central formatApiError(err) helper that translates PostgREST codes (e.g. 42501 → "Permission denied") and falls back to a generic message. Never pass raw err.message directly to UI state or alert().
Medium Severity
Medium Non-Approved Users See Error Screen Instead of Pending-Approval Screen Auth (A07)
src/App.jsx:87–109 · src/store/useAppStore.js:28–41

When a registered but non-approved user logs in, App.jsx calls initializeData() as soon as user is truthy (line 103–106), before the approval status can prevent it. All Supabase fetch calls fail because RLS blocks unapproved users (the is_approved() helper returns false), causing the store to set error = err.message with the raw PostgREST permission-denied message. Because if (error) return <ErrorScreen message={error} /> (line 109) is evaluated before the pending-approval guard (line 113), non-approved users always see the raw error screen — never the intended "Aguardando aprovação" message. This also leaks internal RLS policy error text.

Fix: Guard initializeData() with the approval check:
useEffect(() => { if (user && userProfile?.is_approved) initializeData() }, [user?.id, userProfile?.is_approved])
This ensures data loading only starts for approved users, allowing the pending-approval UI to render correctly.
Medium No Content Security Policy (CSP) Configured Frontend Security (A05)
vite.config.js · (no CSP header anywhere)

The application ships with no Content Security Policy. Without a CSP, any XSS (including the existing html2pdf.js XSS CVE) runs without restriction: can exfiltrate tokens/session data, make API calls to Supabase, or load additional scripts. Vite's dev server also has no CSP. The production deployment target is not configured in this repo (no Nginx/Caddy config, no Vercel/Netlify config visible).

Fix: Add a strict CSP via deployment config or a meta tag:
default-src 'self'; connect-src 'self' https://*.supabase.co; script-src 'self'; style-src 'self' 'unsafe-inline';
Note: html2pdf.js requires 'unsafe-inline' for styles — another reason to upgrade it. Also add X-Frame-Options: DENY and Referrer-Policy: strict-origin.
Medium Admin API Functions Lack Server-Side Role Enforcement (Defense in Depth) Authorization (A01)
src/api/userProfiles.js:9-89

getPendingUsers(), getAllUsers(), updateUserApprovalStatus(), and updateUserRole() only verify that a session exists — they do not check that the caller has role='admin'. A non-admin approved user who calls these functions directly from the browser console (bypassing client-side RBAC guards) will be blocked only by RLS at the database level. While RLS does enforce this correctly, the API layer provides no defense-in-depth: a future misconfiguration of RLS policies would immediately expose full user management to any authenticated user.

Fix: Add an explicit role check at the top of each admin API function by fetching the current user's profile and verifying profile.role === 'admin' before making the Supabase call. This follows the same pattern already used in the auth store's initAuth().
Medium No Admin Demotion UI — Privilege Lock-In Authorization (A01)
src/pages/Admin/Users.jsx:173

The Admin Users page only shows a "Tornar Admin" (Make Admin) button for non-admin users (conditional: u.role !== 'admin'). Once a user is promoted to admin, there is no UI control to demote them back to role='user'. Removing admin privileges requires direct database access (SQL Editor), creating an operational gap: if an admin account is compromised or a user leaves the congregation, admins cannot revoke elevated access through the application itself.

Fix: Add a "Tornar Usuário" (Revoke Admin) button alongside the existing approve/block controls, calling updateUserRole(userId, 'user'). Protect it with a confirmation dialog. Guard against self-demotion (prevent an admin from demoting their own account).

Low Optimistic Logout — Server Session May Persist After Network Failure Auth / Session (A07)
src/store/useAuthStore.js:127–131 · src/api/auth.js:118–127

Logout is intentionally "optimistic": the local state is cleared before the client.auth.signOut() call, and any network error is silently swallowed. If the signOut HTTP request fails, the Supabase refresh token remains valid server-side until it expires naturally, creating a window where the session token could be reused. On a shared/public device, a failed logout could leave an active server-side session.

Fix: Implement a retry on network failure, or at minimum surface a warning to the user when signOut() fails so they know to clear cookies/storage manually. Short Supabase JWT expiry (configured in Supabase dashboard) limits the exposure window.
Low Write Operations Filtered by user_id — Inconsistent With Single-Congregation Model Authorization (A01)
src/api/brothers.js:33-44 · src/api/assignments.js:63-76 · src/api/slots.js:33-59

All UPDATE and DELETE API functions add an .eq('user_id', user.id) filter beyond what RLS already enforces. Per CLAUDE.md, this is a single-congregation app where user_id is traceability, not isolation. This means admin-A cannot edit or delete records created by admin-B (e.g., a different admin's schedules or assignments), breaking the shared-congregation model when multiple admins exist. deleteAssignmentsByWeek() in particular would only clear assignments made by the current admin, silently leaving others' assignments in place.

Fix: Remove .eq('user_id', user.id) from UPDATE/DELETE operations in all data API modules. RLS policies (is_admin()) already restrict writes to admin users only; the additional user_id filter is redundant and harmful. Retain user_id in INSERT for auditing/traceability as designed.
Low JSON Import Lacks Field-Level Validation Input Validation (A03)
src/api/import.js:32–110

validateImportData() checks that the top-level JSON structure is valid and that expected keys are arrays, but does not validate individual records. Item properties are passed directly to API functions (e.g. item.name, item.week_start, item.start_time) without type or length checks. A crafted import file with deeply nested objects, extremely long strings, or non-string values for name fields would propagate through to Supabase, which may produce confusing errors or truncation depending on database constraints.

Fix: Add per-record validation in importDataFromJSON(): ensure name is a non-empty string with a max length (e.g. 255 chars), week_start matches /^\d{4}-\d{2}-\d{2}$/, time fields match HH:MM format, and capacity is a positive integer. Reject records that fail validation instead of attempting insertion.
Low Weak Password Policy — Minimum Length Only Auth (A07)
src/pages/Auth/authValidation.js:22–25 · src/api/auth.js:83-110

Password validation enforces only a minimum length of 8 characters with no complexity requirements (no uppercase, lowercase, digit, or symbol rules). For an administrative scheduling system, weak passwords increase susceptibility to credential-stuffing or brute-force attacks. The corresponding Supabase project password policy settings are not visible in this codebase, so the actual enforcement depends on dashboard configuration.

Fix: Configure a stronger password policy in the Supabase Auth dashboard (minimum 12 characters, require mixed case + digit). Update validatePassword() to reflect the same rules client-side for immediate feedback. Consider adding zxcvbn strength estimation for real-time UX guidance.

Info First-Registration Admin Bootstrap Race Condition (Theoretical) Sec. Misc.
supabase/migrations/20260612000000_create_user_profiles.sql:44-49

The handle_new_user() trigger determines admin status by counting rows in user_profiles WHERE role = 'admin'. If two users register concurrently and both see a zero admin count before either INSERT commits, both would receive role='admin'. For a single-congregation app this is an extremely unlikely scenario, but if exploited (e.g., two rapid signups at initial deployment) it would create two admins. This is informational given the narrow attack window.

Note: For defence, wrap the admin-count check and insert in an advisory lock (pg_advisory_xact_lock) or use a serializable transaction. In practice, initial setup should be a controlled process where the first admin is provisioned manually through the Supabase dashboard.
Info Missing HTTP Security Headers (No Deployment Config Present) Frontend Security (A05)
vite.config.js · (no server config in repo)

No deployment configuration (Nginx, Caddy, Vercel, Netlify) is present in the repository. As a result, the presence or absence of security headers (X-Frame-Options, Referrer-Policy, X-Content-Type-Options, Permissions-Policy) cannot be verified. These are typically set at the reverse-proxy or hosting layer, not within the Vite app itself.

Note: Add deployment configuration that sets at minimum: X-Frame-Options: DENY, X-Content-Type-Options: nosniff, Referrer-Policy: strict-origin-when-cross-origin, and the CSP header noted in finding #7. Supabase CDN / Vercel Edge middleware are good options for a Vite SPA.
